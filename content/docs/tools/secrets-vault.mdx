# Secrets Vault

The Secrets Vault lets you store API keys, tokens, and passwords that your agents can use when calling external APIs. Your agents **never** see the raw credential — they work with opaque handles, and the server injects the real value into HTTP requests behind the scenes.

## How it works

Credential injection follows a three-step flow:

1. **Store a secret** — you add a label, the raw value, and an injection scheme (e.g. Bearer token) in Settings > Secrets Vault.
2. **Agent gets a handle** — when an agent needs to call an external API, it calls the `use_credential` tool with the secret's label. It receives an opaque handle ID — not the raw value.
3. **Server injects the credential** — the agent passes the handle to the `http_request` tool. The server resolves the handle, decrypts the secret, injects it into the correct HTTP header, and sends the request. The raw value never enters the LLM context.

## Adding a secret

Navigate to **Settings > Secrets Vault** and click **Add Secret**.

| Field | Required | Description |
|---|---|---|
| Label | Yes | A short identifier (e.g. `climate-fieldview`, `my-weather-api`). Letters, numbers, hyphens, and underscores only. |
| Description | No | A human-readable note for your reference. |
| Secret Value | Yes | The raw API key, token, or password. Encrypted immediately — never stored in plaintext. |
| Injection Scheme | Yes | How the credential is attached to outgoing HTTP requests (see table below). |
| Allowed Domains | No | Restrict which hosts this credential can be sent to. |
| Allowed Agents | No | Restrict which agents can request a handle for this secret. |

## Injection schemes

| Scheme | HTTP result | Example |
|---|---|---|
| **Bearer Token** | `Authorization: Bearer sk-abc123` | Most REST APIs (OpenAI, Stripe, etc.) |
| **Basic Auth** | `Authorization: Basic base64(user:pass)` | APIs using HTTP Basic authentication |
| **Custom Header** | `X-API-Key: sk-abc123` | APIs that use a named header for auth |
| **Query Parameter** | `?api_key=sk-abc123` | APIs that accept keys in the URL (less common) |

## Domain scoping

You can restrict each secret to specific domains. When allowed domains are set, the `http_request` tool will refuse to send the credential to any other host.

<Callout variant="tip" title="Best practice">
Always scope credentials to the domains that need them. For example, if your API key is for `api.fieldview.com`, set that as the allowed domain. This prevents the credential from being sent anywhere else, even if the agent is tricked by a prompt injection.
</Callout>

## Agent scoping

If you run multiple agents (e.g. Grain Specialist, Weather Bot), you can restrict which agents can request a handle for each secret. When allowed agents are set, only those agents can use the credential — others will receive an error.

Leave the agent list empty to allow all agents.

## Security architecture

<Callout variant="info" title="Encryption details">
Secret values are encrypted at rest using AES-256-GCM with a server-side key. The encrypted ciphertext is never exposed through any API endpoint. Only the masked hint (e.g. `sk-a***23`) is returned to the dashboard.
</Callout>

Key security properties:

- **Ephemeral handles** — credential handles expire after 60 seconds and can only be consumed once.
- **SSRF protection** — the `http_request` tool blocks requests to private IP ranges and requires HTTPS.
- **Response redaction** — if a secret value appears in an API response, it is automatically stripped before the response reaches the LLM.
- **Audit trail** — every handle creation, credential injection, and reveal attempt is logged in `tenant_secret_audit_log` with tenant, agent, and run context.

## What agents can and cannot see

| Visible to agent | Not visible to agent |
|---|---|
| Secret label | Raw secret value |
| Injection scheme (e.g. "bearer") | Encrypted ciphertext |
| Handle ID (opaque reference) | Credentials echoed in HTTP responses |
| Allowed domains list | |

## Break-glass access

<Callout variant="warning" title="Use with extreme caution">
Break-glass access puts the raw secret value directly into the LLM context. This should only be used when credential injection via `http_request` is not possible — for example, when the agent needs to pass an API key inside a request body rather than a header.
</Callout>

The `reveal_secret` tool is an escape hatch for cases where handle-based injection is not sufficient. It is:

- **Disabled by default** — you must explicitly enable it in Settings > Secrets Vault.
- **Approval-gated** — every call requires owner approval before execution.
- **Fully audited** — every reveal is logged with the agent, run, and justification reason.
- **Redacted post-turn** — after the LLM turn completes, revealed values are stripped from stored messages and conversation traces.
- **Rate-limited** — maximum 5 reveals per hour per tenant.

Only secrets with `usage_mode` set to "Break glass" can be revealed. Standard injection-only secrets cannot be revealed even if the tool is enabled.

## FAQ

**Can my agent leak my API key?**
No, under normal operation. The `use_credential` → `http_request` flow keeps the raw value entirely server-side. The agent only sees a handle ID. Even if the agent's conversation is compromised, the attacker gets an opaque handle that expires in 60 seconds, not the actual credential. If you enable break-glass access, the raw value does enter the LLM context for that turn, but it is redacted from stored messages afterward.

**What if I delete a secret mid-conversation?**
Any outstanding handles for that secret will fail when the `http_request` tool tries to consume them. The agent will receive an error message and can inform the user.

**How many secrets can I store?**
There is no hard limit on the number of secrets per workspace. Each secret adds negligible storage overhead.

**Are secrets included in my subscription?**
Yes. The Secrets Vault is included in all Pantheon plans at no additional cost. Only the API calls made through `http_request` count toward your usage metering.
